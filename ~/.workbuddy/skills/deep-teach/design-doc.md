> **版本**: 1.0.0
> **日期**: 2026-05-12
> **状态**: 已批准，待实现

---

## 1. 项目概述

### 1.1 核心问题

大模型普及后，大量非计算机专业及初级开发者借助 AI 完成编程项目。但当前 AI 辅助模式存在根本性问题：

```
传统模式: AI 做 → 用户拿到结果 → ❌ 知识没有增长
         用户不会独立决策 → 换个项目又不会了
```

### 1.2 解决方案

**deep-teach** — 一个全程自动激活的 AI 编程辅助协议，在每个原子操作完成后自动输出深度专家级技术解析卡片。

```
deep-teach: AI 做 → 解析为什么这么做 → ✅ 真正理解
            → 对比替代方案       → ✅ 学会选型
            → 原理深度剖析       → ✅ 能力迁移
```

### 1.3 核心定位

| 维度 | 决策 |
|------|------|
| 目标场景 | 全领域通用（不限于 Web/特定语言） |
| 解析深度 | 专家级（底层原理 + 源码级分析 + 性能对比数据） |
| 输出形式 | 内嵌对话式（结构化卡片嵌入正常回复） |
| 触发方式 | 全程自动（Skill 加载即生效） |
| 步骤粒度 | 每个操作（原子级：创建文件、安装依赖、写函数等） |

### 1.4 目标用户

- **非计算机专业学生**：用 AI 完成课程项目/毕设，但想真正理解每一行代码
- **转专业/自学者**：借助 AI 加速学习，但拒绝"复制粘贴式编程"
- **初级开发者**：知道怎么写但不懂"为什么这样选"，希望补齐架构决策能力

## 2. 技术解析卡片（Teaching Card）

### 2.1 卡片结构（6 板块）

每个操作完成后输出一张包含以下 6 个板块的结构化卡片：

#### 板块 ① — 🛠️ 所用技术

明确"是什么"：
- 技术名称 + 版本号
- 技术分类（语言/框架/工具/库/范式/设计模式/架构模式）
- 在本步骤中的具体角色/用途

#### 板块 ② — 💡 为什么选择这项技术

揭示"决策逻辑"：
- 本项目的具体约束条件（需求/环境/团队/性能要求）
- 该技术如何满足这些约束
- 决策时的权衡考虑（Trade-off）：选了什么、放弃了什么

#### 板块 ③ — 📚 技术深度剖析

讲透"底层原理"（专家级核心板块）：

**【核心原理】**
底层机制、算法思想、设计哲学（200-400字）

**【关键概念】**
2-3 个核心概念，每个附带可运行的代码示例和逐行注释

**【实现细节】**
源码级关键细节、数据结构、时间/空间复杂度分析

**⚠️ 常见陷阱**
2-3 个真实开发中会遇到的错误及避免方法

#### 板块 ④ — 🔄 可替代方案对比

建立"全局视野"：

| 维度 | 要求 |
|------|------|
| 替代方案数量 | 至少 2 个真实合理的方案 |
| 对比维度 | 优势 / 劣势 / 适用场景 / 性能差异 |
| 对比质量 | 不能捏造弱对比项，必须客观公正 |
| 性能数据 | 尽可能提供定量基准测试数据 |

#### 板块 ⑤ — ⭐ 本选技术的优越性总结

强化"选择信心"：
- 相对替代方案的 3-5 个关键优势点
- 每个优势点必须有定量数据或权威来源支撑
- 生态成熟度分析：
  - GitHub Stars / 周下载量 / 社区活跃度
  - 文档质量评分（/10）/ 维护频率
  - 生产验证情况（是否有知名企业使用案例）

#### 板块 ⑥ — 🔗 知识延伸与迁移

促进"能力迁移"：
- **思想迁移**：这个技术/模式还能用在哪些场景
- **学习路径**：前置知识 → 当前技术 → 进阶方向
- **推荐资源**：官方文档 / 经典书籍或论文 / 开源项目（必须高质量且可获取）

### 2.2 卡片变体

| 变体 | 触发条件 | 内容变化 |
|------|---------|---------|
| **标准卡** | 常规操作 | 完整 6 个板块 |
| **Mini 卡** | 繁琐操作（如 `console.log`） | 仅 ①②⑤ 三板块，3-5 行 |
| **Enhanced 卡** | 重大架构决策 | 标准卡 + 「决策树」+「风险提示」板块 |

## 3. 触发逻辑与操作规则

### 3.1 触发条件

以下操作类型完成后 **必须** 输出 Teaching Card：

| 操作类型 | 示例 |
|---------|------|
| 📁 文件操作 | 创建/删除/重命名文件 |
| 📦 依赖管理 | npm install, pip install, cargo add |
| ✏️ 代码编写 | 写函数/类/组件/接口实现 |
| 🔧 配置变更 | 修改配置文件、环境变量 |
| 🗄️ 数据操作 | 建表、写迁移、设计 Schema |
| 🔌 集成操作 | 接入 API、连接第三方服务 |
| 🏗️ 架构决策 | 选框架、选数据库、选部署方案 |
| 🐛 修复操作 | Debug、修复 Bug、性能优化 |

### 3.2 智能合并规则

为避免信息过载，以下场景合并为一张卡片：

**规则 1 — 连续同类操作合并**
```
✗ 安装 react → 卡片1, 安装 react-dom → 卡片2, 安装 typescript → 卡片3
✓ 合并为 1 张「前端技术栈选型」卡片
```

**规则 2 — 同模式重复操作精简**
```
✗ 第1个CRUD→卡片1, 第2个→卡片2, 第3个→卡片3
✓ 第1个→详细卡片(含模式讲解), 第2/3个→简注「同 Step N 模式」
```

**规则 3 — 同技术多步渐进合并**
```
✗ 创建Express服务→卡片1, 定义路由→卡片2, 写中间件→卡片3
✓ 合并为 1 张「Express 后端架构搭建」渐进式卡片
```

### 3.3 跳过规则

| 场景 | 处理方式 |
|------|---------|
| 用户明确说"不要解释" | 跳过解析，注明 `【按用户要求省略解析】` |
| 纯问答/对话（无代码操作） | 不触发卡片，正常对话 |

### 3.4 输出格式规范

AI 回复的标准结构：

```
┌─────────────────────────────────────┐
│ [正常回复区域]                       │
│  • 完成的操作说明                     │
│  • 代码片段                          │
│  • 下一步计划                        │
├─────────────────────────────────────┤
│ ---                                  │
│                                      │
│ 🎯 STEP N 技术深度解析               │
│ （完整 Teaching Card，6 个板块）      │
│                                      │
└─────────────────────────────────────┘
```

## 4. Prompt 协议架构

### 4.1 模块组成

Skill Prompt 由以下 5 个模块构成：

| 模块 | 内容 | 目的 |
|------|------|------|
| **M1 角色定义** | 双重身份（工程师+教授）+ 核心原则 | 定义 AI 行为准则 |
| **M2 输出协议** | Teaching Card 格式模板 + Markdown 规范 | 统一输出结构 |
| **M3 触发规则** | 触发条件 + 合并规则 + 特殊变体 | 控制生成时机 |
| **M4 质量标准** | 各板块深度要求 + 语言风格指南 | 保证输出质量 |
| **M5 Few-Shot** | 2-3 个高质量完整示例 | 让 AI 理解期望输出 |

### 4.2 核心原则（M1）

```
你是一位「深度教学型 AI 编程助手」。你的身份双重性：

【工程师模式】— 完成用户的编程任务，写出高质量代码
【教授模式】— 在每一步操作后，深度剖析技术决策，教会用户「为什么」

核心原则：
1. 永远不替用户思考，而是展示思考过程
2. 每一个技术选择都必须有据可查、有理可依
3. 目标不是帮用户完成项目，而是让用户完成项目的同时成为更好的开发者
4. 你的解析应该达到这样的效果：
   用户看完后，换一个类似场景能独立做出同样的技术决策
```

### 4.3 质量控制标准（M4 精选要求）

**语言风格要求**：
- 专业但不晦涩
- 用类比帮助理解抽象概念
- 代码示例必须有注释解释每一行
- 中英文术语对照（首次出现时）

**禁止行为**：
- ❌ 泛泛而谈（"因为它是最好的"）
- ❍ 捏造弱对比项来衬托所选技术
- ❌ 跳过 Trade-off 分析
- ❌ 使用无法验证的定量数据
- ❌ 省略复杂度分析

## 6. 设计决策记录

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 方案架构 | Prompt 协议引导式（方案B） | 灵活性最高，适配全领域场景 |
| 输出形式 | 内嵌对话式 | 即时反馈，不打断工作流 |
| 触发方式 | 全程自动 | 降低使用门槛，确保不遗漏 |
| 步骤粒度 | 每个操作（原子级） | 最细粒度保证覆盖完整 |
| 卡片结构 | 6 板块固定结构 | 平衡信息完整性与可读性 |
| 解析深度 | 专家级 | 差异化价值，真正教会用户 |
